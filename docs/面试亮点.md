# 股票交易系统 — 面试亮点

> 基于 Spring Boot 3 + React 18 的全栈模拟股票交易系统

---

## 一、项目概况

| 维度 | 内容 |
|------|------|
| 技术栈 | Java 17, Spring Boot 3.3.5, Spring Data JPA, Spring Security, JWT, Redis, MySQL 8.0 |
| 前端 | React 18 + TypeScript + Vite + Ant Design 5 + ECharts + pinyin-pro |
| 数据源 | 腾讯行情 API（`qt.gtimg.cn`）+ 新浪股票列表 API，本地 ±5% 模拟降级 |
| 架构 | 单体应用（前后端分离），Maven 构建，前端 Vite 打入 `src/main/resources/static/` |

## 二、核心亮点

### 1. 事务一致性：买入卖出原子操作

**文件**: `TradingServiceImpl.java`

- 使用 `@Transactional` 保证下单、扣款、更新持仓、写成交记录在同一个数据库事务中
- 买入时 **加权平均成本算法**：`(原持仓市值 + 新买入金额) / 总持仓数量`
- 卖出时 **已实现盈亏计算**：`卖出金额 - 卖出数量 × 加权平均成本`
- 面试话术："我通过 Spring 声明式事务确保资金和持仓的强一致性，同时用加权平均成本计算持仓成本，这是真实交易系统的标准做法。"

### 2. 实时行情：腾讯 API + 优雅降级

**文件**: `MarketDataServiceImpl.java`

- 优先调用 **腾讯行情 API**（`qt.gtimg.cn`）获取 A 股实时价格，`~` 分隔文本格式，解析极快
- 替代不稳定的东方财富接口（间歇性空响应 + UA 屏蔽），腾讯接口稳定且无需代理
- API 不可达时（网络问题、非交易时段）自动降级到 **本地随机模拟**（±5% 涨跌）
- Redis 缓存行情（10s TTL），Redis 不可用时自动回退 MySQL
- 面试话术："行情模块我换过两版——东方财富不稳定被 UA 屏蔽，最终切到腾讯，响应毫秒级，API 挂了还有本地模拟和 MySQL 双层兜底。"

### 3. 全量 A 股导入 + 看板管理

**文件**: `StockMasterServiceImpl.java`, `StockMasterController.java`, `MarketDataController.java`

- 启动时从 **新浪股票列表 API** 分页拉取 5200+ 只 A 股代码和中文名，存入 `stock_master` 表
- 看板（watchlist）机制：用户可自由增删关注的股票，`@Scheduled` 只刷新看板中的股票
- 新增 `POST /api/market-data/add?code=xxx` 和 `/remove?code=xxx` 管理看板
- 面试话术："全量 A 股一次性入库，实时刷新只针对用户关注的股票。不是不能刷 5000 只，而是没必要——关注多少刷多少，按需分配带宽和算力。"

### 4. 拼音搜索 + 前端缓存

**文件**: `StockSearch.tsx`, `pinyin-pro` 库

- 前端安装 `pinyin-pro`，支持 **股票代码 / 中文名 / 拼音首字母** 三种搜索方式
- 搜 "mt" → 匹配"贵州茅台"，搜 "600" → 匹配所有 600 开头的沪市股票
- 全量股票列表在页面加载时从后端拉一次，缓存在浏览器内存，搜索纯前端完成，零后端压力
- AutoComplete 组件实时筛选，选中即加入看板
- 面试话术："拼音搜索用 `pinyin-pro` 库在前端算，5530 条数据约 200KB JSON，一次加载终身使用。搜索体验和东方财富一样快，而且不费后端。"

### 5. 无状态 JWT 认证

**文件**: `SecurityConfig.java`, `JwtAuthenticationFilter.java`

- 基于 jjwt 库生成 HMAC-SHA256 签名的 JWT Token
- 自定义 `OncePerRequestFilter` 拦截请求，解析 Token 并注入 `SecurityContext`
- 公开接口白名单（登录、注册、行情、搜索、排行榜、Swagger），其余需认证
- 面试话术："无状态认证让系统天然支持水平扩展，不需要 session 同步，每次请求独立鉴权。"

### 6. 风控与业务逻辑分离

**文件**: `RiskControlServiceImpl.java`

- 独立的风控校验模块，和 `TradingService` 完全解耦
- 校验项：股票可交易性、单笔数量上限（1w 股）、单笔金额上限（100w）、余额充足性、持仓充足性
- 所有阈值通过 `application.yml` 配置，无需改代码即可调整
- 面试话术："风控是金融系统的生命线。我把校验逻辑抽成独立 Service，既保证了 TradingService 的单一职责，也方便后续扩展更多风控规则（如日内撤单次数、IP 频率限制）。"

### 7. AOP 统一交易日志

**文件**: `TradeLogAspect.java`

- 通过 `@Around` 切面拦截 `placeOrder`，无侵入记录每笔交易
- 成功日志：userId、orderId、股票代码、买卖方向、价格、数量、耗时
- 失败日志：userId、原因、请求参数
- 面试话术："用 AOP 做统一审计日志，核心业务代码零侵入。如果后续需要接入 ELK 或发交易消息到 MQ，只需要改切面而不动业务逻辑。"

### 8. 前端实时行情 + 动画

**文件**: `useQuotes.ts`, `Dashboard.tsx`, `Trading.tsx`

- 自定义 Hook `useQuotes` 每 5 秒轮询行情接口，自动管理价格变化闪烁动画（↑↓ 箭头）
- ECharts 绘制折线图（Dashboard）/ K 线图（Trading），暗黑金融主题
- Ant Design 5 + `#0f172a` 暗色背景，12px 圆角卡片，红绿色专业配色
- 面试话术："前端通过自定义 Hook 封装轮询逻辑，组件只关心渲染。价格闪烁动画提升用户体验，`useRef` 记录旧值做 diff 避免不必要的重渲染。"

### 9. 多级缓存架构

- **一级**：Redis 缓存行情（10s 过期）
- **二级**：MySQL 持久化
- **降级路径**：当 Redis 宕机，直接从 MySQL 读取；定时刷新也会更新 Redis
- 面试话术："缓存不是银弹。我设计的是 cache-aside 模式 + 定时刷新，即使用了 Redis 也仍然保证 MySQL 是最终数据源，缓存丢了不影响业务。"

## 三、可能被问到的面试题

### Q: 这个项目是真实上线项目还是 Demo？
A: 模拟交易系统 Demo，技术方案参考了真实券商系统（账户体系、持仓管理、风控校验）。数据源接入了腾讯和新浪的公开 API，但交易为模拟，无真实资金。

### Q: 为什么用 `@Transactional` 而不是分布式事务？
A: 当前为单体应用，所有操作在同一数据库事务内完成，Spring 声明式事务足够保证 ACID。如果未来拆分为微服务（如交易服务、账户服务分离），会引入 Saga 或 TCC 模式。

### Q: 如何处理并发下单？
A: 当前利用数据库行级锁 + 乐观锁。如果是高并发场景，可以用 Redis 分布式锁 + 消息队列削峰，交易请求先入队再批量处理。

### Q: 行情延迟怎么处理？
A: 腾讯接口延迟约 3-10 秒，演示场景可接受。生产环境应接入券商 Level-2 行情（毫秒级），用 WebSocket 推送而非定时轮询。

### Q: 前端缓存 5000 多只股票会不会卡？
A: 不会。JSON 约 200KB，`pinyin-pro` 计算一次拼音约 0.1ms，搜索在 Web Worker 里做也不影响主线程。实际测试输入即出，无感知延迟。

### Q: 为什么换掉了东方财富接口？
A: 实测东方财富有几个问题： 1）单股接口频繁返回空数据 2）批量接口字段映射不统一 3）会屏蔽 Java 默认的 User-Agent。腾讯 `qt.gtimg.cn` 格式固定，无 UA 校验，延迟更低。

### Q: 前端为什么不直接调行情 API？
A: 跨域限制 + 数据清洗 + 缓存。前端请求经过后端代理，后端在中间层做 Redis 缓存、API 降级、格式统一，前端只关心渲染。这是 BFF（Backend For Frontend）模式。

### Q: 前端性能优化做了哪些？
A: 1. React.memo 避免无关组件重渲染 2. ECharts 增量更新（`setOption` 而不是销毁重建） 3. 轮询间隔 5s 平衡实时性和性能 4. Ant Design 按需加载 5. 拼音搜索纯前端缓存，零后端请求

## 四、演示流程

1. 首页搜索框输入股票代码/中文名/拼音首字母（如"mt"→贵州茅台）
2. 选中股票加入看板，实时行情自动刷新
3. 注册/登录账号
4. 在交易页买入股票
5. 在持仓页查看资产汇总（总资产、持仓市值、收益率）
6. 排行榜查看所有用户收益率排名
7. 重复买入卖出观察收益率变化
