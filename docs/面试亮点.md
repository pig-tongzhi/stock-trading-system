# 股票交易系统 — 面试亮点

> 基于 Spring Boot 3 + React 18 的全栈模拟股票交易系统

---

## 一、项目概况

| 维度 | 内容 |
|------|------|
| 技术栈 | Java 17, Spring Boot 3.3.5, Spring Data JPA, Spring Security, JWT, Redis, MySQL 8.0 |
| 前端 | React 18 + TypeScript + Vite + Ant Design 5 + ECharts |
| 架构 | 单体应用（前后端分离），Maven 多模块，内嵌 H2（开发）/ MySQL（生产） |
| 部署 | JAR 包部署，前端构建后打入 `src/main/resources/static/` |

## 二、核心亮点

### 1. 事务一致性：买入卖出原子操作

**文件**: `TradingServiceImpl.java`

- 使用 `@Transactional` 保证下单、扣款、更新持仓、写成交记录在同一个数据库事务中
- 买入时 **加权平均成本算法**：`(原持仓市值 + 新买入金额) / 总持仓数量`
- 卖出时 **已实现盈亏计算**：`卖出金额 - 卖出数量 × 加权平均成本`
- 面试话术："我通过 Spring 声明式事务确保资金和持仓的强一致性，同时用加权平均成本计算持仓成本，这是真实交易系统的标准做法。"

### 2. 双重数据源：实时行情 + 优雅降级

**文件**: `MarketDataServiceImpl.java`

- 优先调用 **东方财富公开 API**（`push2.eastmoney.com`）获取 A 股实时行情
- 批量接口 `ulist.np/get` 一次请求携带全部股票，避免多次 HTTP 调用
- API 不可达时（网络问题、非交易时段、接口变更）自动降级到 **本地随机模拟**（±5% 涨跌）
- Redis 缓存行情数据（10s TTL），Redis 不可用时自动回退 MySQL
- 面试话术："行情模块是系统的第一道门面，我设计了三级降级策略：东方财富 → 本地模拟 → MySQL 兜底，保障演示永不空数据。"

### 3. 无状态 JWT 认证

**文件**: `SecurityConfig.java`, `JwtAuthenticationFilter.java`

- 基于 jjwt 库生成 HMAC-SHA256 签名的 JWT Token
- 自定义 `OncePerRequestFilter` 拦截请求，解析 Token 并注入 `SecurityContext`
- 公开接口白名单（登录、注册、行情、排行榜、Swagger），其余需认证
- 面试话术："无状态认证让系统天然支持水平扩展，不需要 session 同步，每次请求独立鉴权。"

### 4. 风控与业务逻辑分离

**文件**: `RiskControlServiceImpl.java`

- 独立的风控校验模块，和 `TradingService` 完全解耦
- 校验项：股票可交易性、单笔数量上限（1w 股）、单笔金额上限（100w）、余额充足性、持仓充足性
- 所有阈值通过 `application.yml` 配置，无需改代码即可调整
- 面试话术："风控是金融系统的生命线。我把校验逻辑抽成独立 Service，既保证了 TradingService 的单一职责，也方便后续扩展更多风控规则（如日内撤单次数、IP 频率限制）。"

### 5. AOP 统一交易日志

**文件**: `TradeLogAspect.java`

- 通过 `@Around` 切面拦截 `placeOrder`，无侵入记录每笔交易
- 成功日志：userId、orderId、股票代码、买卖方向、价格、数量、耗时
- 失败日志：userId、原因、请求参数
- 面试话术："用 AOP 做统一审计日志，核心业务代码零侵入。如果后续需要接入 ELK 或发交易消息到 MQ，只需要改切面而不动业务逻辑。"

### 6. 前端实时行情 + 动画

**文件**: `useQuotes.ts`, `Dashboard.tsx`

- 自定义 Hook `useQuotes` 每 5 秒轮询行情接口，自动管理价格变化闪烁动画
- 使用 ECharts 绘制折线图 / K 线图
- Ant Design 5 暗黑主题，专业金融风格 UI
- 面试话术："前端通过自定义 Hook 封装轮询逻辑，组件只关心渲染。价格闪烁动画提升用户体验，代码上使用 `useRef` 记录旧值做 diff，避免不必要的重渲染。"

### 7. 多级缓存架构

- **一级**：Redis 缓存行情（10s 过期）
- **二级**：MySQL 持久化
- **降级路径**：当 Redis 宕机，直接从 MySQL 读取；定时刷新也会更新 Redis
- 面试话术："缓存不是银弹。我设计的是 cache-aside 模式 + 定时刷新，即使用了 Redis 也仍然保证 MySQL 是最终数据源，缓存丢了不影响业务。"

### 8. 全栈面试题覆盖

| 方向 | 对应代码 |
|------|---------|
| Spring 事务 | `@Transactional placeOrder` 涉及 4 张表的原子操作 |

| 外部 API | RestTemplate 调用东方财富接口 |
| 前端框架 | React + Hooks + TypeScript + Vite |
| UI 组件 | Ant Design 5 暗黑主题 |
| 数据可视化 | ECharts 折线图 / K 线图 |

## 三、可能被问到的面试题

### Q: 这个项目是真实上线项目还是 Demo？
A: 这是一个模拟交易系统 Demo，技术方案参考了真实券商系统的设计（账户体系、持仓管理、风控校验）。数据源接入了东方财富的公开行情 API，但交易为模拟，无真实资金。

### Q: 为什么用 `@Transactional` 而不是分布式事务？
A: 当前为单体应用，所有操作在同一数据库事务内完成，Spring 声明式事务足够保证 ACID。如果未来拆分为微服务（如交易服务、账户服务分离），会引入 Saga 或 TCC 模式。

### Q: 如何处理并发下单？
A: 当前利用数据库行级锁 + 乐观锁。如果是高并发场景，可以用 Redis 分布式锁 + 消息队列削峰，交易请求先入队再批量处理。

### Q: 行情延迟怎么处理？
A: 东方财富接口延迟约 3-10 秒，演示场景可接受。生产环境应接入券商 Level-2 行情（毫秒级），用 WebSocket 推送而非定时轮询。

### Q: 前端为什么不直接调东方财富？
A: 跨域限制 + Token 鉴权。前端请求经过后端代理，后端可以在中间层做缓存、数据清洗、权限校验，这是 BFF 模式（Backend For Frontend）的实践。

### Q: 前端性能优化做了哪些？
A: 1. React.memo 避免无关组件重渲染 2. ECharts 增量更新（`setOption` 而不是销毁重建） 3. 轮询间隔 5s 平衡实时性和性能 4. Ant Design 按需加载

## 四、演示流程

1. 打开首页查看 8 只股票的实时行情
2. 注册/登录账号
3. 在交易页选择股票进行买入操作
4. 在持仓页查看资产汇总（总资产、持仓市值、收益率）
5. 排行榜查看所有用户收益率排名
6. 重复买入卖出观察收益率变化
